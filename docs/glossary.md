# Glossary CI/CD Demo Project

Данный глоссарий содержит определения терминов и понятий, используемых в проекте.  
Он предназначен для разработчиков, технических писателей, DevOps-инженеров и всех
участников команды.

Глоссарий является частью подхода **Docs as Code** и поддерживается в актуальном
состоянии вместе с кодовой базой.

---

## API (Application Programming Interface)

Программный интерфейс взаимодействия между системами.  
В рамках проекта API представляет собой HTTP REST-интерфейс, предоставляемый backend-приложением на Flask.

---

## Backend

Серверная часть приложения, отвечающая за:

- обработку бизнес-логики;
- валидацию входных данных;
- выполнение вычислений;
- формирование и возврат JSON-ответов клиенту.

В проекте реализован с использованием **Python + Flask**.

---

## Build

Процесс подготовки приложения к запуску или распространению.  

Может включать:

- установку зависимостей;
- компиляцию;
- сборку артефактов (например, Docker-образа или frontend-бандла).

---

## CI (Continuous Integration)

Непрерывная интеграция — практика автоматической проверки изменений кода при каждом коммите или pull request.

В проекте CI включает:

- установку зависимостей;
- запуск автоматических тестов;
- сборку Docker-образа;
- проверку документации.

---

## CI/CD Pipeline

Последовательность автоматизированных шагов, выполняемых системой CI/CD.  
Pipeline описывается в виде конфигурационных файлов (например, GitHub Actions workflows).

---

## CORS (Cross-Origin Resource Sharing)

Механизм безопасности браузера, ограничивающий HTTP-запросы между разными доменами.

Для корректной работы frontend и backend, расположенных на разных доменах, backend
должен возвращать специальные HTTP-заголовки CORS.

---

## Container (Контейнер)

Изолированная среда выполнения приложения, включающая:

- код приложения;
- зависимости;
- системные библиотеки;
- настройки окружения.

В проекте используются контейнеры Docker.

---

## Docker

Платформа для контейнеризации приложений.  
Позволяет упаковать приложение и все его зависимости в единый переносимый образ.

---

## Docker Image (Docker-образ)

Шаблон контейнера, содержащий:

- файловую систему приложения;
- зависимости;
- инструкции запуска.

Docker-образы в проекте публикуются в **Docker Hub**.

---

## Docker Hub

Облачный registry для хранения и распространения Docker-образов.

---

## Docs as Code

Подход к ведению документации, при котором:

- документация хранится в репозитории вместе с кодом;
- используется контроль версий (Git);
- изменения проходят через pull request;
- документация проверяется и разворачивается автоматически.

---

## Environment (Окружение)

Контекст, в котором выполняется приложение:

- Local — локальная машина разработчика;
- CI — окружение GitHub Actions;
- Production — платформа Render.

---

## Frontend

Клиентская часть приложения, работающая в браузере пользователя.

В проекте реализована на **React** и отвечает за:

- пользовательский интерфейс;
- отправку запросов к backend API;
- отображение результатов и ошибок.

---

## GitHub Actions

Сервис автоматизации от GitHub, используемый для реализации CI/CD.

Позволяет запускать workflows при:

- push;
- pull request;
- создании тегов.

---

## GitHub Pages

Сервис GitHub для хостинга статических сайтов.  
В проекте используется для публикации документации, собранной с помощью MkDocs.

---

## Markdown

Лёгкий язык разметки для написания документации.  
Используется для всех файлов в директории `docs/`.

---

## markdownlint

Инструмент для проверки качества Markdown-документов:

- стиль;
- структура;
- соблюдение соглашений форматирования.

---

## MkDocs

Инструмент для генерации статических сайтов документации из Markdown-файлов.

Используется для:

- сборки документации;
- навигации;
- публикации на GitHub Pages.

---

## Pipeline Failure

Ситуация, при которой один или несколько шагов CI/CD завершаются с ошибкой.  
Требует анализа логов и исправления проблемы до продолжения деплоя.

---

## Pull Request (PR)

Механизм внесения изменений в основную ветку репозитория.

Для технического писателя pull request является:

- основным источником информации об изменениях;

- точкой контроля актуальности документации.

---

## Render

Облачная платформа для деплоя web-приложений и Docker-контейнеров.

Используется для хостинга backend-приложения проекта.

---

## REST API

Архитектурный стиль построения API, основанный на HTTP-методах и ресурсах.

---

## Semantic Versioning (SemVer)

Схема версионирования вида `MAJOR.MINOR.PATCH`, используемая для управления релизами.

---

## Static Site

Веб-сайт, состоящий из заранее сгенерированных HTML-файлов, не требующий серверной логики.

Документация проекта является статическим сайтом.

---

## Webhook

Механизм уведомления внешней системы о событии посредством HTTP-запроса.

В проекте используется webhook для триггера деплоя в Render.

---

## Workflow

Набор шагов автоматизации, описанный в GitHub Actions и выполняемый при наступлении событий.

---

## Unified API Response

Единый формат ответа API, используемый для всех эндпоинтов проекта, включающий:

- признак успеха;
- данные;
- информацию об ошибке.

---

## Version Control System (VCS)

Система управления версиями кода и документации.  
В проекте используется **Git**.

---

## Production

Окружение, в котором приложение доступно конечным пользователям.

---

## Documentation Debt

Накопленная разница между фактическим поведением системы и её документацией.
Является аналогом технического долга.
